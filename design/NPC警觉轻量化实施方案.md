# NPC警觉轻量化实施方案

> 日期：2026-06-05  
> 适用范围：当前 Godot 项目中的碎片 NPC 对话系统，优先覆盖 #0762「颜色的葬礼」，后续可复用于 0001/0002 等碎片。  
> 目标：保留“玩家身份可能暴露”的叙事张力，但不实现重型通缉、追捕、全城封锁系统。

---

## 一、结论

本项目不建议继续实现完整的“全局警觉/通缉系统”。推荐将警觉降级为：

**NPC 独立戒备/信任系统。**

它只影响 NPC 的对话态度、信息释放深度、局部拒绝交互和恢复关系，不驱动全城追捕、不锁死主线、不强制玩家逃跑。

当前代码中最适合承接该系统的是：

- `scripts/fragment/npc_controller.gd`
- `scripts/systems/npc_rag_retriever.gd`
- `scripts/ui/chat_dialogue.gd`
- `scripts/globals/game_manager.gd`

当前不建议继续扩展：

- `scripts/systems/wanted_system.gd`

该文件可以暂时保留，但 MVP 阶段不接入主流程。

---

## 二、为什么不做重型警觉

### 2.1 重型警觉的成本

文档中曾设计过：

- 全局 6 级警觉
- NPC 互相传递玩家情报
- 全城封锁
- NPC 追捕玩家
- 躲藏 60 秒归零
- 警觉模式跨碎片差异化

这些内容技术上可做，但会带来四类成本：

| 成本 | 说明 |
|---|---|
| 行为 AI 成本 | NPC 需要可靠寻路、追逐、回避、复位、卡住处理。 |
| 关卡设计成本 | 每张地图都要设计追捕路径、躲藏点、禁区、失败恢复点。 |
| 叙事调试成本 | 玩家问错一句话可能导致剧情锁死，需要大量例外规则。 |
| 体验风险 | 游戏会从情感探索变成“猜哪些话不能问”的惩罚体验。 |

### 2.2 轻量警觉的收益

轻量警觉只保留核心叙事价值：

- 玩家不是无风险地审问 NPC。
- NPC 有自己的边界、恐惧和防御反应。
- “迷路旅人”身份不只是背景设定，而是玩家说话时需要维持的身份。
- 送礼、帮助、颜色觉醒可以让 NPC 重新信任玩家。

这与 #0762 的核心体验更一致：玩家不是躲避敌人，而是在修复人与人之间失去的情感连接。

## 三、系统定义

### 3.1 核心概念

| 概念 | 定义 |
|---|---|
| NPC 警觉值 | 单个 NPC 对玩家的戒备程度，范围 `0-100`。 |
| NPC 警觉阶段 | 由警觉值映射出的态度阶段。 |
| 身份怀疑标记 | NPC 是否已经开始怀疑玩家“不属于这里”。 |
| 安全行为 | 能降低警觉的玩家行为或话语。 |
| 敏感话题 | 会提高警觉的词汇、问题或行为。 |
| 恢复动作 | 送对物品、完成颜色觉醒、说出符合身份的话等。 |

### 3.2 阶段设计

沿用 `npc_controller.gd` 中已有的 `NPCAlertPhase`：

| 阶段 | 数值范围建议 | 玩家感受 | NPC表现 | 是否阻断主线 |
|---|---:|---|---|---|
| `TRUSTING` 信任 | 0-19 | 正常交流 | 愿意回答，能给提示 | 否 |
| `CAUTIOUS` 谨慎 | 20-39 | 轻微变冷 | 回答变短，偶尔回避 | 否 |
| `SUSPICIOUS` 怀疑 | 40-59 | 明显不信任 | 反问、含糊、不给深层信息 | 不阻断，只降低提示 |
| `ALARMED` 警觉 | 60-79 | 关系紧张 | 拒绝危险话题，要求玩家离开 | 暂时阻断该 NPC 的深层信息 |
| `HOSTILE` 敌对 | 80-94 | 本轮失败 | 结束对话、短暂逃离或不再回应 | 不锁死，可恢复 |
| `SHUTDOWN` 封闭 | 95-100 | 极端失信 | 完全拒绝交互 | 仅用于极端输入，不建议常见触发 |

设计原则：

1. 警觉可以让线索变难拿，但不应该永久锁死碎片。
2. 每个高警觉状态必须有恢复路径。
3. 玩家应该从 NPC 反应中理解“我问错了方向”，而不是只看到数值惩罚。

## 四、实现范围

### 4.1 MVP 必做

| 模块 | 状态 | 处理 |
|---|---|---|
| NPC 独立警觉值 | 已有 | 继续使用 `npc_suspicion`。 |
| NPC 警觉阶段 | 已有 | 继续使用 `npc_alert_phase`。 |
| 警觉持久化 | 已有 | 继续使用 `GameManager.npc_state_cache`。 |
| 敏感词触发 | 已有函数但未接入 | 在 `send_player_message()` 中接入。 |
| 安全行为降警觉 | 已有函数但未接入 | 在 `send_player_message()` 中接入。 |
| RAG 警觉上下文 | 已有 | 继续注入 `alert_context`。 |
| 弹窗反馈 | 已有 | 保留，但文案需克制。 |

### 4.2 MVP 不做

| 功能 | 原因 |
|---|---|
| 全局 WantedSystem 接入 | 当前主体验不需要，且几乎没有调用点。 |
| NPC 全城共享情报 | 成本高，且容易造成剧情锁死。 |
| 追捕/躲藏/逃跑倒计时 | 会改变游戏类型。 |
| 警觉条常驻 UI | 容易让玩家把它当战斗/潜行数值。 |
| LLM 作为主要裁判 | 不稳定，模型输出会影响数值，难调试。 |

## 五、触发规则

### 5.1 三层检测模型

警觉触发分三层：

1. **硬风险词**
2. **NPC 专属敏感话题**
3. **安全行为/修复话题**

本地规则负责判定，LLM 负责表现。

### 5.2 硬风险词

硬风险词指 NPC 在当前世界观下不应该听到的外部概念。

建议词表：

```gdscript
const GLOBAL_FORBIDDEN_TOPICS := [
	"AI", "npc", "NPC", "代码", "程序", "游戏", "模拟", "虚拟",
	"源印", "天枢", "天枢公司", "万象", "溯光者", "归源计划",
	"织女", "太一", "冥府", "冥府协议"
]
```

触发建议：

| 类型 | 警觉变化 | 说明 |
|---|---:|---|
| 直接外部词 | `+15` | 如“AI”“NPC”“代码”。 |
| 世界真相词 | `+20` | 如“万象”“溯光者”“归源计划”。 |
| 源印/公司词 | `+25` | 如“源印”“天枢”“冥府协议”。 |

注意：即使玩家是否定句，也应触发。

例：

> “我不是来找源印的。”

对 NPC 来说，“源印”本身就是危险词。触发警觉合理。

### 5.3 NPC 专属敏感话题

继续使用 `_get_alert_profile()` 中的 `hot_topics`。

示例：

| NPC | 敏感点 | 设计意图 |
|---|---|---|
| 老霍 | 火、以前、没有颜色、你在怕什么 | 他对火和愤怒有执念。 |
| 阿莲 | 你还好吗、不一样、镇子不对劲 | 她温柔但容易封闭。 |
| 老唐 | 炉子、镇上少了什么、过去 | 他用乐观压住不安。 |
| 老崔 | 铁门、死人/活人、呼吸、冥府 | 他最敏锐，也最害怕。 |
| 薇拉 | 琴声、等谁、那天、零时 | 她用沉默保护记忆。 |
| 冯婶 | 登记簿、柜台、房钱、说谎 | 她是旅店秩序的守门人。 |

触发建议：

| 类型 | 警觉变化 |
|---|---:|
| 普通敏感话题 | `+8` |
| 反复追问同一话题 | `+12` |
| 直接追问核心秘密 | `+18` |
| 触及该 NPC 绝对红线 | `+25` |

### 5.4 安全行为

每个 NPC 必须有至少 3 个安全行为关键词。

安全行为不是“拍马屁”，而是玩家维持旅人身份、尊重 NPC 边界的表现。

示例：

| NPC | 安全行为关键词 | 降警觉 |
|---|---|---:|
| 老霍 | 手艺、铁器、锤子沉、买东西 | `-6` |
| 阿莲 | 买花、花很漂亮、谢谢、慢慢来 | `-6` |
| 老唐 | 饿了、面包香、帮忙、揉面 | `-8` |
| 老崔 | 沉默、点头、我不问了、花 | `-8` |
| 薇拉 | 等、听、坐一会儿、不打扰 | `-8` |
| 冯婶 | 累了、歇一晚、付房钱、谢谢 | `-6` |

### 5.5 恢复动作

| 恢复动作 | 效果 |
|---|---|
| 给予正确物品 | `-15`，已有 `ALERT_GIVE_ITEM`。 |
| 触发该 NPC 颜色觉醒 | 该 NPC 警觉归零。 |
| 其他颜色觉醒后再对话 | 若警觉不高于 `SUSPICIOUS`，每轮可 `-2`。 |
| 离开一段时间 | 自然衰减，已有 `_process_alert_decay()`。 |
| 道歉/承认越界 | 可作为通用安全行为，`-5`。 |

### 5.6 反复追问惩罚

建议新增一个轻量记忆：

```gdscript
var _last_sensitive_topic: String = ""
var _repeat_sensitive_count: int = 0
```

规则：

- 如果连续命中同一个敏感词，第二次起额外 `+4`。
- 达到 3 次，NPC 进入本轮拒绝状态。
- 离开对话或安全行为命中后清零。

这能避免玩家机械刷问同一句。

## 六、代码接入方案

### 6.1 接入点

文件：

`scripts/fragment/npc_controller.gd`

函数：

`send_player_message(message: String)`

当前流程中，玩家输入被记录后直接进入 RAG/LLM。建议在收集 `game_state` 之前接入本地警觉判定。

建议修改位置：

```gdscript
func send_player_message(message: String) -> void:
	print("[NPC] %s 收到: \"%s\"" % [npc_name, message])
	_llm_last_input = message

	# 新增：先由本地规则更新警觉。
	check_safe_action(message)
	check_player_input_for_alert(message)

	ChatDatabase.log_message(npc_kb_id, "player", message, npc_alert_phase, npc_suspicion)
	...
```

原因：

1. 本地规则先更新 `npc_suspicion`。
2. `_collect_game_state()` 随后拿到最新 `alert_level`。
3. RAG prompt 立刻获得正确的 `alert_context`。
4. LLM 回复会自然变冷、回避或拒绝。

### 6.2 裁判与表现分离

推荐职责划分：

| 模块 | 职责 |
|---|---|
| `check_player_input_for_alert()` | 本地规则裁判：是否加警觉。 |
| `check_safe_action()` | 本地规则裁判：是否降警觉。 |
| `get_alert_context_for_rag()` | 将当前阶段转化为 NPC 心理状态。 |
| `NPCRagRetriever.assemble_prompt()` | 注入警觉上下文，让 LLM 表现态度。 |
| `_analyze_llm_response_for_alert()` | 可选辅助，不做主裁判。 |

### 6.3 调整 LLM 反馈分析

当前 `_analyze_llm_response_for_alert()` 会根据 LLM 回复长度、警告词、回避词再改警觉。

建议保留，但降低权重：

| 当前行为 | 建议 |
|---|---|
| 极短回复 `+15` | 改为 `+4` |
| 警告词 `+25` | 改为 `+8` |
| 回避词 `+8/+18` | 改为 `+3/+6` |
| 反问 `+12` | 改为 `+4` |
| 主动谈真相 `-8` | 保留或改为 `-5` |

原因：LLM 输出不是稳定裁判源。它应该微调，而不是决定主要数值。

### 6.4 WantedSystem 处理

短期处理：

1. 不删除 `scripts/systems/wanted_system.gd`。
2. 不在新碎片中主动调用 `wanted_system.add_suspicion()`。
3. `fragment_base.gd` 中的 `WantedIndicator` 可保留，但默认隐藏。
4. 后续若 0001 需要“合规度”，可以重命名或另做 `ComplianceSystem`，不要复用通缉文案。

建议在 README 或技术 TODO 中标注：

> `WantedSystem` 为早期全局通缉原型，MVP 暂不接入主流程。实际警觉逻辑以 `npc_controller.gd` 的 per-NPC suspicion 为准。

## 七、数据结构建议

### 7.1 NPC Alert Profile

当前 `_get_alert_profile()` 已返回：

```gdscript
{
	"thresholds": {...},
	"hot_topics": [...],
	"personality": "...",
	"safe_actions": [...],
	"trust_modifier": 0.0
}
```

建议扩展为：

```gdscript
{
	"thresholds": {...},
	"hot_topics": [...],
	"safe_actions": [...],
	"absolute_red_lines": [...],
	"recovery_hints": [...],
	"trust_modifier": 0.0
}
```

字段说明：

| 字段 | 用途 |
|---|---|
| `absolute_red_lines` | 命中后直接进入高警觉，如“闯进墓园铁门后”。 |
| `recovery_hints` | 高警觉后给玩家的修复方向。 |

### 7.2 恢复提示

当 NPC 进入 `ALARMED` 或 `HOSTILE` 时，系统应给一个角色化恢复提示，而不是只惩罚。

示例：

```gdscript
func _get_recovery_hint() -> String:
	match npc_kb_id:
		"baker":
			return "（老唐避开你的视线。）如果你只是饿了，就说饿了。别问那些让人睡不着的事。"
		"gravekeeper":
			return "（老崔把铁门关小了一点。）你可以站远一点。安静一点。别再问门后面。"
	return ""
```

### 7.3 状态持久化

继续使用：

```gdscript
GameManager.save_npc_state(scene_name, npc_kb_id, global_position, npc_suspicion, npc_alert_phase, doubts_player_identity)
```

注意：

- 警觉应跟随 NPC 和场景保存。
- 颜色觉醒后允许清空该 NPC 的警觉。
- 重新挑战碎片时，是否保留警觉要按设计决定。

当前 `GameManager.reset_fragment()` 注释写的是保留 NPC 信任/警觉状态。若体验上过于严厉，可以改成：

- 重玩碎片保留对话历史；
- 重置 NPC 警觉到 `0`；
- 老画家信任保留。

该点需后续试玩验证。

## 八、UI 与反馈

### 8.1 不推荐常驻警觉条

原因：

- 玩家会把游戏理解为“潜行数值管理”。
- 过度显性会破坏叙事沉浸。
- 当前项目核心是对话和情感修复，不是风险条管理。

### 8.2 推荐轻量反馈

保留现有 `ChatDialogue.show_alert_popup()`，但文案改得更叙事化。

建议：

| 变化 | UI 文案 |
|---|---|
| `+5` 以下 | 不弹或只轻微提示 |
| `+6 ~ +14` | “老唐的笑容淡了一点” |
| `+15 ~ +24` | “阿莲开始回避这个问题” |
| `+25+` | “老崔不再相信这是普通闲聊” |
| 降警觉 | “对方稍微放松了一些” |

避免显示：

- “警觉 +18”
- “当前警觉值 43”
- “阶段：SUSPICIOUS”

数值用于调试，不直接面向玩家。

### 8.3 开发调试显示

调试模式可以显示：

```text
[Debug] 老唐 suspicion=42 phase=SUSPICIOUS
```

但正式 UI 不显示。

## 九、与 RAG 的关系

### 9.1 RAG 输入

`_collect_game_state()` 当前会传入：

```gdscript
state["alert_level"] = int(npc_suspicion)
state["alert_context"] = get_alert_context_for_rag()
```

这是正确方向。

### 9.2 RAG 输出原则

RAG 和 LLM 只负责：

- 让 NPC 变得谨慎、回避、沉默；
- 控制信息释放；
- 保持角色口吻；
- 不承认外部世界词汇。

RAG 不负责：

- 判断玩家是否违规；
- 直接增减警觉数值；
- 决定碎片是否失败。

### 9.3 Knowledge JSON 配合

每个 NPC 的 JSON 中建议保留：

- `security` chunks：处理越狱/外部词。
- `alert_required`：用于某些“高警觉后才说”的内容。
- `memory_stage`：用于进度门控。

但不要把所有警觉逻辑塞进 JSON。JSON 是知识，不是规则引擎。

## 十、碎片模式差异

### 10.1 #0762：NPC 独立警觉

| 项 | 方案 |
|---|---|
| 系统名 | 戒备/信任 |
| 表现 | NPC 变冷、回避、拒绝深问 |
| 恢复 | 送物品、颜色觉醒、符合身份的表达 |
| 是否全局共享 | 否 |
| 是否封城 | 否 |

### 10.2 #0001：合规度

0001 是公司培训/监控碎片，不建议叫“警觉”。

| 项 | 方案 |
|---|---|
| 系统名 | 合规度 / 监控指数 |
| 表现 | 公司 NPC 脚本化、权限不足、系统重置 |
| 恢复 | 回到流程、按规则完成训练 |
| 是否敌意 | 否，是冷冰冰的系统限制 |

技术上可以复用部分数值结构，但 UI 和文案必须完全不同。

### 10.3 0002-3333：按需启用

后续碎片不必每个都强行启用完整警觉。

建议每个碎片只问一个问题：

> 这个碎片中，“玩家不属于这里”是否是核心体验？

如果不是，就只保留安全层，不显示警觉反馈。

### 10.4 4096：关闭

终局不需要伪装。关闭警觉。

## 十一、实施步骤

### 阶段 1：接入现有函数

目标：让现有关键词检测真正生效。

修改：

- `scripts/fragment/npc_controller.gd`

任务：

1. 在 `send_player_message()` 中调用：

```gdscript
check_safe_action(message)
check_player_input_for_alert(message)
```

2. 确认调用顺序在 `_collect_game_state()` 之前。
3. 测试输入“源印”“天枢”“为什么没有颜色”能改变 `npc_suspicion`。

验收：

- 玩家说风险词后，RAG prompt 中 `alert_level` 变高。
- NPC 回复态度变冷。
- 弹窗出现但不影响继续游玩。

### 阶段 2：降低 LLM 反向判定权重

目标：避免模型输出导致警觉大幅波动。

修改：

- `_analyze_llm_response_for_alert()`

任务：

1. 降低所有正向加警觉值。
2. 保留少量自动修正能力。
3. 确保本地关键词是主裁判。

验收：

- 同一句玩家输入不会因为 LLM 回复不同而产生大幅数值差异。

### 阶段 3：优化 NPC 配置

目标：让每个 NPC 的警觉变化符合角色。

修改：

- `_get_alert_profile()`

任务：

1. 为每个 NPC 补足 `safe_actions`。

## 十三、风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 关键词误伤 | 玩家正常问话被惩罚 | 降低普通敏感词权重，只对硬风险词重罚。 |
| 玩家看不到原因 | 只弹“警觉增加” | 改成 NPC 反应文案。 |
| 高警觉难恢复 | 玩家感觉被锁死 | 每个 NPC 至少 3 个恢复入口。 |
| LLM 输出不稳定 | 同样输入结果差异大 | 本地规则做主裁判，LLM 分析降权。 |
| 系统过于数值化 | 玩家开始刷关键词 | 正式 UI 隐藏数值，只保留角色反馈。 |

## 十四、最终体验目标

玩家应该感受到：

> “我不是在操控一个警觉条。  
> 我是在和一个害怕、迟疑、会保护自己的人说话。”

系统成功的标志不是玩家学会避开关键词，而是玩家开始自然地改变说话方式：

- 从“告诉我源印在哪”变成“我只是想知道你在等什么”。
- 从“你们是 AI 吗”变成“你有没有觉得这里少了什么”。
- 从“为什么没有颜色”变成“这朵花对你重要吗”。

这就是本项目需要保留警觉的原因：它不是惩罚系统，而是让玩家学会尊重 NPC 边界的叙事装置。
