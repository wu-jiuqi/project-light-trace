# 溯光计划 — 物品设计需求规范 V1.0

> **用途**：可复用的物品设计规范模板。定义"物品是什么、如何分类、如何描述、有什么美术标准、技术如何对接"——适用于任意碎片的物品设计，不包含具体碎片的物品清单。
> **"物品"定义**：游戏中所有需要美术资源的物件——建筑、场景模块、地面铺装、道具、解密物品、UI组件、FX特效、材质纹理、环境陈设等。
> **适用范围**：全项目所有碎片的物品设计（含 Core 全局资产）
> **最后更新**：2026-06-04
> **参考基础**：`style_bible.md`、`asset_manifest_format.md`、`art_spec_地图美术规格.md`、`12碎片叙事规划_V1.0.md`
> **配套文档**：`物品设计审查文档.md`（审查标准）、各碎片设计文档中的物品清单章节

## 一、设计流程

### 1.1 碎片级工作流

当为一个新碎片设计物品时，按以下步骤推进：

```
① 通读碎片设计文档
   提取所有场景区域、建筑、道具、解密物品、特效、暗线植入物
   ↓
② 按分类体系（§二）对每个物品归类
   建筑/地面 → environment；可交互小物 → prop；演出 → fx；界面 → ui
   ↓
③ 按单物品描述规范（§三）逐物品填写
   每件物品必须给出：ID、名称、分类、描述、尺寸、材质、优先级
   ↓
④ 按类型的专项设计指引（§四）检查各类型物品的特殊要求
   建筑："场所身份锚点" / 解密道具："视觉独特感" / FX：触发逻辑
   ↓
⑤ 按美术规格（§五）逐项自查
   材质族合规、裁切合规、阴影层级、色值协调、Shader穿透（如适用）
   ↓
⑥ 按技术对接（§七）注册 Manifest、命名文件、确认 Godot 集成方式
   ↓
⑦ 按交付物清单（§八）打包提交审查
```

### 1.2 设计约束

- 物品描述面向"美术人员直接制作"——使用具体的像素值、HEX色值、材质ID
- 每个物品必须有明确的**视觉锚点**：缩放到 32px 后仍可辨识的核心特征
- 建筑类物品必须采用"场所身份锚点转化为建筑结构"方向（参照 §四·4.1）
- 所有物品描述与 `style_bible.md` 冲突时，以 `style_bible.md` 为准
- **禁止模棱两可**：不能用"大约""灰色""旧旧的感觉""纸片风格"等模糊描述

## 二、物品分类体系

### 2.1 分类枚举

| 分类 | 英文 ID | 说明 | 典型示例（仅供理解，非具体物品） |
|------|---------|------|------|
| 环境-建筑 | `environment` | 建筑立面、房屋模块、围墙、门等不可进入的外部结构或大型固定结构 | 旅店立面、神社本殿、工坊外观 |
| 环境-地面 | `environment` | 地面铺装、道路模块、广场铺地、室内地板 | 碎石路、石板广场、木地板 |
| 环境-室内 | `environment` | 室内陈设、家具、固定装置 | 柜台、锻造台、烤炉、墓碑、书架 |
| 道具 | `prop` | 可交互/可收集的小型独立物品 | 日志、钥匙、花束、齿轮、日晷 |
| 特效 | `fx` | 演出特效、觉醒拼贴、源印、裂缝 | 觉醒光效、颜色裂缝、星图粒子 |
| UI | `ui` | 界面组件 | 对话框、按钮、背包界面、星图面板 |
| 材质 | `material` | 独立纹理资源（Core全局） | 牛皮纸纹理、描图纸纹理 |
| 角色 | `character` | 角色纸偶（已有独立设计规范，物品清单仅记录 NPC 持有的独立道具） | NPC工具、配饰、手持物 |

### 2.2 物品优先级

| 优先级 | 定义 | 判定标准 |
|--------|------|---------|
| **P0** | 垂直切片必需，阻挡核心玩法 | 没有此物品→玩家无法完成碎片核心解密或关键交互 |
| **P1** | 碎片完整版必需 | 没有此物品→碎片可通关但缺少重要场景/道具/演出 |
| **P2** | 全量版或优化项 | 没有此物品→碎片功能完整，缺少的是细节陈设或可选装饰 |

## 三、单物品描述规范

### 3.1 必填字段

每个物品必须包含以下字段。**任何一项缺失或模糊 → 打回补充。**

| # | 字段 | 格式要求 | 示例（正确） | 示例（错误） |
|---|------|---------|-------------|-------------|
| 1 | **ID** | `[scope]_[category]_[short_name]`，全小写英文+下划线 | `id0762_env_square_statue` | `雕像_1` ❌ |
| 2 | **名称** | 中文，1-8字，准确反映物品身份 | "白色雕像" | "一个东西" ❌ |
| 3 | **分类** | 从 §2.1 枚举中选择 | `environment` | `场景物品` ❌ |
| 4 | **描述** | 完整段落。包含：形态、材质质感、颜色（含HEX）、关键视觉特征、在场景中的位置/状态。面向美术人员。 | 见 §3.2 | 见 §3.3 |
| 5 | **尺寸** | 像素值（如 `48×64px`）或 tile 数（如 `3 tiles 宽`），两者都给出最佳 | `高48px（含底座），底座24×20px` | "中等大小" ❌ |
| 6 | **材质** | 从 `kraft`/`gray_card`/`book_page`/`fiber_paper`/`tracing_paper`/`foil` 中选择1-2个 | `gray_card`+`book_page` | "纸的" ❌ |
| 7 | **优先级** | `P0` / `P1` / `P2` | `P1` | "重要" ❌ |

### 3.2 描述字段写作要求

描述段落必须覆盖以下信息维度（不要求固定顺序，但不能遗漏）：

1. **形态与结构**：是什么形状？有什么组成部分？对称/不对称？
2. **材质与质感**：看起来像什么材料？表面光滑/粗糙/有纹理？是剪裁的还是撕边的？
3. **颜色**：给出至少1个 HEX 色值。给出关键视觉区域的颜色对比关系（如"墙面比普通建筑稍亮"）
4. **磨损/使用痕迹**（如适用）：哪里有磨损？是什么造成的？有多明显？
5. **视觉锚点**：缩到32px后，靠什么特征辨认这个物品？
6. **叙事关联**（如适用）：此物品承载什么剧情信息？它的视觉状态如何暗示这个信息？
7. **动态状态**：静止的？在动的？（如"窗帘微微飘动"）有多个状态吗？（如"觉醒前灰色/觉醒后蓝色"）

### 3.3 禁止的描述方式

| 禁止写法 | 为什么禁止 | 正确替代 |
|---------|-----------|---------|
| "灰色" | 灰度范围太广（#333~#CCC都是灰色） | 给出 HEX，如 `#808080`，并描述冷暖倾向 |
| "大约3 tiles" | 美术无法据此定尺寸 | 去掉"大约"，给精确值 `3 tiles（48px）` |
| "旧旧的感觉" | 不可量化 | 描述磨损的具体位置和程度："右下角有深色烫痕" |
| "纸工风格" | 不指定材质族 | 给定材质 ID：`kraft`+`gray_card` |
| "类似XX"（与已有物品比较） | 无自有特征定义 | 描述该物品独特的视觉锚点 |
| "合适大小" | 无参照系 | 给像素值或 tile 数 |

## 四、按类型的设计指引

### 4.1 建筑类物品（environment - 建筑）

建筑类物品是碎片中最重要、最复杂的物品类型。除了满足 §三 的所有必填字段外，还需遵守：

**A. "场所身份锚点转化为建筑结构"原则**

每座建筑的视觉设计必须将"这个建筑是什么地方"的身份锚点转化为建筑结构元素。不是给普通房子贴个招牌——是让建筑本身"变成"它的功能。

| 建筑类型 | 身份锚点 | 转化方向（示例） |
|---------|---------|----------------|
| 旅店 | 住客登记簿 | 旧书页→立面墙体 / 书脊→屋脊线 / 装订带→结构固定件 / 折页→屋檐 |
| 面包房 | 食谱/烤炉/麦穗 | 食谱书页→屋脊结构 / 烤炉砖拱→门楣造型 / 麦穗→墙面装饰纹 |
| 花店 | 花架/花盆/矢车菊 | 花架纵横线条→立面骨架 / 花瓣轮廓→顶饰 / 花盆→窗沿 |
| 神社 | 鸟居/和歌/注连绳 | 鸟居造型→入口 / 和歌纸片→屋瓦层叠 / 纸垂→屋檐坠饰 |
| 工坊 | 齿轮/蓝图/钟摆 | 齿轮→立面圆形窗 / 蓝图网格→墙面划分 / 钟摆→门廊吊饰 |

**B. 建筑必须保留的可读性底线**

以下元素不能因"概念化"而被牺牲：
- 门的位置和宽度——玩家必须一眼看出从哪里进入
- 窗的位置——暗示室内空间的存在
- 屋檐——界定建筑的视觉顶边界
- 入口朝向——门朝向哪个方向（决定玩家接近路线）
- 建筑在地面上的占地面积轮廓——碰撞盒的视觉依据

**C. 建筑物品特有字段**

| 额外字段 | 要求 |
|---------|------|
| **占地面积** | 以 tiles 为单位，如 `宽 8 tiles × 高 6 tiles` |
| **入口描述** | 门在哪个面？多宽？有什么特殊门饰？ |
| **屋顶描述** | 坡顶/平顶？材质？颜色？有什么屋顶特征？ |
| **墙面材质使用** | 主材质 + 哪些区域使用了不同材质 |

### 4.2 地面类物品（environment - 地面）

**A. 道路/广场模块**

- 必须定义模块边界和拼接接口——相邻模块如何对齐？
- 如果有变体（如 straight/corner/t_junction），必须在描述中列出所有变体
- 拼接边缘必须是直线（不能是不规则撕边——会导致接缝可见）
- 内部纹理可以是不规则拼贴，但外轮廓不能破坏对齐

**B. 室内地板**

- 与室外地面有明显色值差异（室内通常更暗）
- 如果室内有"特殊区域"（如锻造台周围被铁屑覆盖的地面），描述其边界和渐变

**C. 地面物品特有字段**

| 额外字段 | 要求 |
|---------|------|
| **拼接接口** | 模块的哪几条边需要与其他模块对齐？直线还是曲线？ |
| **变体列表** | 需要哪些方向变体？（straight/edge/corner/t_junction/cross） |
| **磨损/踩踏痕迹** | 频繁使用的路径上地面是否有变浅/变光滑的区域？ |

### 4.3 道具类物品（prop）

**A. 解密道具**

核心解密道具是玩家必须找到并与之交互的物品。它们的视觉设计有一条铁律：

> **"在常态中隐隐不同"**——正常 gameplay 视距下，解密道具不能大声叫"看我看我"，但必须有微弱的视觉差异让细心的玩家注意到。

实现方式：
- 与其他同类物品有细微色差（如灰阶世界中有一朵花"灰得不一样"）
- 放置位置稍微"不自然"（如放在不该出现的地方）
- 有独特的轮廓或比例（如唯一一本封面有裂痕的书）
- 如果碎片有 Shader 穿透系统，解密道具可以是微弱的颜色穿透点

**B. 可收集文件/书籍/日志**

- 使用 `book_page` 材质
- **严格禁止**生成可读文字——只用模糊的书写/印刷痕迹暗示"这上面有字"
- 封面/封底需要独立于内页的视觉特征（如"书脊有磨损""封面有烫金压痕"）
- 如果书是翻开的，描述可见的页面布局（是列表？段落？草图？）

**C. 手持工具/配饰（NPC关联物品）**

- 这些物品在角色美术规格中有描述——但在物品清单中需要独立列出，因为它们需要独立制作
- 描述必须与角色美术规格中的色值和形态一致
- 如果是 Shader 穿透标记（如围裙上的烫痕），需要同时注册对应的 `_sp_mask.png`

**D. 道具物品特有字段**

| 额外字段 | 要求 |
|---------|------|
| **交互方式** | 玩家如何与此物品交互？（拾取/阅读/推动/对话触发/靠近观察） |
| **是否可携带** | 进入背包？留在场景中？ |
| **是否有多个状态** | 如"觉醒前灰色/觉醒后蓝色""打开前/打开后""完整/碎裂" |
| **Shader穿透** | 如适用，标注穿透区域和穿透饱和度 |

### 4.4 特效类物品（fx）

**A. 觉醒/演出FX**

- 必须明确触发条件（什么事件触发此特效开始播放？）
- 必须描述持续时间或循环行为（一次性/持续/脉冲式闪烁/渐进过渡）
- FX 纸片的叠加方式：叠加到场景上（Additive）/替换（Replace）/覆盖（Overlay）
- 描图纸（`tracing_paper`）用于光效和透明过渡；银箔纸（`foil`）用于高价值和源印强调

**B. Shader驱动的视觉变化**

- 如果某些视觉变化由运行时 Shader 参数控制（而非独立 PNG），在物品描述中标注"运行时Shader"
- 仍需描述视觉目标效果，即使实现方式不是独立美术资产

**C. FX物品特有字段**

| 额外字段 | 要求 |
|---------|------|
| **触发条件** | 什么事件触发？时间点？ |
| **持续时间** | 一次性/循环/渐进/脉冲（如脉冲给周期） |
| **叠加方式** | Additive/Replace/Overlay |
| **影响范围** | 单个物品/单个区域/场景级/全域 |

### 4.5 UI类物品（ui）

- UI 组件为 Core 全局资产，不按碎片拆分（除碎片专属界面外）
- 遵循统一的设计语言方向（由 `ref_ui_components` 黄金参考定义）
- 所有 UI 面板严格禁止生成可读文字——内容区为空白旧书页/矩形负空间
- UI 物品描述中标注该组件在游戏中的用途（如"对话系统""星图界面""背包界面"）

## 五、美术规格统一要求

> 本节为所有碎片通用的不可协商标准。与 `style_bible.md` 冲突处以后者为准。

### 5.1 材质族分配规则

每个物品必须从以下材质族中选择主要材质（1个）和次要材质（0-2个）：

| 材质 ID | 适用物品类型 | 使用限制 |
|---------|------------|---------|
| `kraft` | 木制品、家具、门框、普通建筑、箱子、柜台、工作台 | 避免大面积高饱和色；允许纵向木纹和钉孔 |
| `gray_card` | 石材、天空、灰布、雕像、地面铺装、金属物品 | 允许压痕、磨损和轻度风化；灰阶碎片的建筑主体材质 |
| `book_page` | 书籍、文件、信件、招牌、UI面板内容区 | **禁止生成可读文字**。仅保留模糊书写/印刷痕迹（如模糊的列表结构、段落痕迹） |
| `fiber_paper` | 衣物、布帘、围裙、毯子、窗帘 | 纤维纹理不能盖住主体识别度；撕边可用于布帘下摆 |
| `tracing_paper` | 雾、裂缝、源印、光效、觉醒FX、星图碎片 | 控制透明度，不能影响交互可读性；通常作为叠加层使用 |
| `foil` | 源印、星图、高价值标记 | **严格克制**。单物品使用面积不超过10%；仅用于六色源印和少数高价值视觉强调 |
| `fastener` | 胶带、线绳、铆钉（固定件，非独立材质） | 每个主体最多选择两类固定件；用量克制，不能让立面变成杂物拼贴 |

### 5.2 禁止使用的风格

以下为全项目硬性禁止（不可变规则）：

- ❌ 光滑矢量边缘（必须保留不规则手工裁切感）
- ❌ 塑料、橡胶或写实3D玩具质感
- ❌ 将普通插画简单叠加纸张纹理后当作纸工资产（纸工资产必须从头就是纸工）
- ❌ 每张资产使用完全不同的纸纹和光源方向（全项目统一左上光源）
- ❌ 纯黑 `#000000` 和纯白 `#FFFFFF`（灰阶碎片中雕像特许 #F0F0F0+#FFFFFF 高光除外）
- ❌ 建筑退化为无法分辨入口、层级和碰撞边界的纯概念图

### 5.3 裁切与阴影

- **裁切**：所有物品外轮廓保留轻微不规则剪刀切边。**建筑类物品不允许撕边**（撕边仅用于异常演出和少数布帘下摆）
- **纸层厚度**：使用窄暗边（1-2px）表达纸层厚度，不使用厚重写实倒角
- **阴影颜色**：偏暖灰或冷灰，**不使用纯黑**
- **阴影偏移按层级分配**：

| 层级 | 物品类型 | 阴影偏移 | 示例 |
|------|---------|---------|------|
| L0 | 地面贴图 | 0–1px | 地面石板、道路纹理 |
| L1 | 矮道具 | 1–2px | 花盆、面团、钥匙 |
| L2 | 建筑/大型家具 | 2–4px | 建筑立面、柜台、锻造台 |
| L3 | 前景遮挡 | 4–8px | 前景枯树、门框前景 |
| L4 | 悬浮/UI | 单独定义 | 星图碎片、源印、对话框 |

### 5.4 光源统一

- **光源方向**：全局统一左上方 → 阴影统一向右下方偏移
- **光色**：暖白或中性白，不使用彩色光（Shader FX 除外）
- **高光**：仅用于金属、玻璃和源印（`foil`），普通纸材不使用高光

### 5.5 Shader 穿透标记规范（仅适用于有灰阶 Shader 的碎片）

如果碎片使用了全局灰阶 Shader（如 #0762），需要为特定物品的颜色残留设计穿透：

| 规则 | 说明 |
|------|------|
| 每个物品最多 1 个穿透区域 | 过多会削弱稀缺感 |
| 穿透饱和度 10%–25% | 源印载体可达 100% |
| 需要独立 `_sp_mask.png` | 与物品同尺寸的二值图，白色=穿透区域 |
| 穿透元素必须与该区域的情感/颜色锚点一致 | 不能随机选择穿透部位 |
| 碎片中穿透物品总数建议 | 每个碎片 3–7 个穿透物品（含 NPC 穿透） |

## 七、技术对接

### 7.1 文件命名规范

```
assets/papercraft/<scope>/<category>/<item_id>.png
```

- `<scope>`：`core` 或碎片 ID（如 `id0762`）
- `<category>`：`environment` / `prop` / `fx` / `ui` / `material` / `character` / `mask`
- `<item_id>`：全小写英文 + 下划线，如 `market_street_ground_straight`

Shader 穿透 Mask：
```
assets/papercraft/<scope>/<category>/<item_id>_sp_mask.png
```

### 7.2 命名规范细则

| 规则 | 说明 |
|------|------|
| 仅使用小写字母、数字、下划线 | 不含中文、空格、连字符、特殊符号 |
| 前缀 | `<scope>_<category>_` 开头，如 `id0762_env_`、`core_ui_` |
| 物品简称 | 描述物品的最短英文短语，如 `square_statue`、`inn_facade` |
| 变体 | 在简称后追加，如 `market_street_ground_straight`、`market_street_ground_corner` |
| Mask | 与主物品同 ID + `_sp_mask` 后缀 |

### 7.3 Asset Manifest 注册格式

每个物品必须在对应碎片的 `assets/papercraft/manifests/[scope].json` 中注册。格式参照 `asset_manifest_format.md`：

```json
{
  "id": "[scope]_[category]_[short_name]",
  "path": "assets/papercraft/fragments/[scope]/[category]/[short_name].png",
  "category": "[environment|prop|fx|ui|material|character|mask]",
  "role": "[物品功能的中文简短描述]",
  "status": "planned",
  "generation": "[isolated_asset|reference_edit|ui_component|effect_composite|manual_mask]",
  "requiredFor": "[vertical_slice|[scope]_full|global_ui]",
  "notes": "[视觉锚点和限制，与需求规范中的描述摘要一致]",
  "variants": ["[变体1]", "[变体2]"]
}
```

**Manifest 字段约束**：

| 字段 | 约束 |
|------|------|
| `id` | 必须与需求规范中的物品 ID 一致 |
| `category` | 必须使用合法枚举值 |
| `status` | 新物品统一为 `planned` |
| `generation` | 建筑/道具→`isolated_asset`；UI→`ui_component`；FX→`effect_composite`；Mask→`manual_mask`；基于参考图编辑→`reference_edit` |
| `requiredFor` | Core资产→`global_ui` 或 `vertical_slice`；碎片资产→`[scope]_full` 或 `vertical_slice` |
| `variants` | 仅当物品确实有多个变体时填写 |

### 7.4 状态流转

```
planned → candidate → review → ready
```

- `planned`：仅存在于需求规范和 Manifest 中，尚无 PNG
- `candidate`：已生成候选 PNG，尚未完成审查
- `review`：已完成初步清理，等待最终验收
- `ready`：正式入库，可接入 Godot

### 7.5 Godot 场景集成原则

| 物品类型 | Godot 节点 | 注意事项 |
|---------|-----------|---------|
| 建筑模块 | `Sprite2D` → 挂载到 `TileMapLayer` 或独立节点 | 碰撞盒只覆盖可通行/不可通行边界 |
| 地面模块 | `TileMapLayer` | 使用 `16×16 px/tile` 逻辑网格 + `4x` 纸片纹理 |
| 道具 | `Area2D` + `Sprite2D` | 用于可交互检测；非交互道具可只用 `Sprite2D` |
| FX | `Sprite2D` + Shader 动画参数 | 确认叠加方式（Additive/Replace/Overlay） |
| 所有物品 | 脚底锚点统一在画布底边中央 | 建筑类放在底边对应地面位置 |
