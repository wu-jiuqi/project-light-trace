# 溯光计划 — 剧情审查需求文档

> **用途**：每完成一个碎片/剧情的设计后，将此文档 + 该剧情相关全部材料交给 Agent，Agent 将按此规范逐项审查并输出标准化审查结果。  
> **版本**：V1.0  
> **最后更新**：2026-06-04  
> **审查对象**：单个碎片剧情（如 fragment_0002.md）或一组连续的碎片剧情

---

## 一、审查输入：需要提供的材料

在发起审查时，请将以下材料一并提供给 Agent：

### 必选材料

| # | 材料 | 说明 |
|---|------|------|
| 1 | **当前碎片完整设计文档** | 如 `fragment_XXXX.md`，包含解密逻辑、NPC设定、暗线植入、流程时间线等 |
| 2 | **该碎片的所有 NPC 设计文档** | 如 `npc_XX.md`，含角色设定、对话风格、System Prompt、安全层规则 |
| 3 | **该碎片的技术代码**（如有） | 如 `fragment_XXXX.gd`、`fragment_XXXX_state.gd`、NPC 控制器脚本 |
| 4 | **12碎片叙事规划**（全局参考） | `12碎片叙事规划_V1.1.md`，用于检查暗线递进和跨碎片关联 |

### 可选材料（如有则提供，提高审查精度）

| # | 材料 | 说明 |
|---|------|------|
| 5 | **前置碎片的设计文档** | 用于检查衔接连续性 |
| 6 | **美术规格文档** | Style Bible、角色美术规格、地图规格，用于检查视觉元素可实现性 |
| 7 | **开发进度文档** | 最新 Day N 开发进度，用于检查技术可行性 |
| 8 | **世界观设计文档** | 用于检查设定一致性 |

## 二、审查维度总览

Agent 必须逐项完成以下 **7 个维度** 的审查。每个维度均有具体检查项和判定标准。

```
维度1：解密逻辑合理性    →  谜题是否可解？是否存在矛盾或死锁？
维度2：前置剧情衔接      →  与前置碎片的故事线是否连续无断裂？
维度3：暗线植入审查      →  三条暗线的植入点是否正确、足够、不突兀？
维度4：剧情元素可实现性  →  场景/NPC行为/特效在技术栈下是否可实现？
维度5：全局一致性审查    →  与其它文档的设定、数值是否一致？
维度6：设计-代码对齐     →  （如有代码）代码实现是否符合设计文档？
维度7：玩家体验安全审查  →  是否存在软锁/死档/认知过载/情感过载？
```

## 三、逐维度检查清单

### 维度 1：解密逻辑合理性

> 目标：确认谜题在逻辑层面自洽，不存在矛盾或不可解路径。

**1.1 核心解密逻辑逐步推演**

- [ ] 用实际数据推演解密全流程（不是读文档描述，而是自己代入玩家视角走一遍）
- [ ] 验证每一步的输入→输出是否唯一确定
- [ ] 如果涉及数学/排序/约束，用实际参数验证解的唯一性
- [ ] 检查是否存在"玩家即使没有全部线索也能猜对"的情况（谜题泄露）
- [ ] 检查是否存在"玩家收集了全部线索但仍无法推理"的情况（信息不足）

**1.2 边界情况**

- [ ] 玩家不按设计顺序收集线索时，推理是否仍可完成？
- [ ] 玩家中途退出并重新进入碎片时，状态是否正确恢复？
- [ ] 如果谜题允许多次尝试，错误次数上限是多少？达到上限后如何处理？

**1.3 提示系统**

- [ ] 是否有分层提示机制（如 L1→L4）？
- [ ] 最低层提示（卡关兜底）是否能确保玩家最终通过？
- [ ] 提示的触发条件是否合理（不会太早剧透，也不会太晚导致玩家已放弃）？

**1.4 逻辑矛盾检测**

逐条列出碎片中所有**声明性规则**（如"A等于B"、"如果X则Y"、"只有Z能触发W"），然后检查：
- [ ] 规则之间是否存在直接矛盾？
- [ ] 规则是否隐含矛盾（A→B, B→C, C→非A）？
- [ ] 例外情况是否覆盖了所有边缘状态？

**输出格式**：
```
| 审查项 | 结论 | 详情 |
|--------|------|------|
| 逐步推演 | ✅/⚠️/🔴 | … |
| 边界情况 | … | … |
| 提示系统 | … | … |
| 逻辑矛盾 | … | … |

综合评级：🟢通过 / 🟡基本通过(需澄清N处) / 🔴存在关键问题
```

### 维度 2：前置剧情衔接

> 目标：确认当前碎片与前置碎片的故事线无缝衔接，没有断裂、跳跃或矛盾。

**2.1 通关解锁衔接**

- [ ] 本碎片的前置解锁条件是什么？前一个碎片是否在结束时明确告知/暗示了本碎片的存在？
- [ ] 如果前一个碎片结束时给出了多个选择（如0001→0002或0003），本碎片的进入方式是否与之一致？

**2.2 暗线连续性**

分别检查三条暗线（A: AI不是敌人 / B: 公司阴谋 / C: 平行世界）：
- [ ] 本碎片对该暗线的推进/种子/引爆是否与前一个碎片形成递进（不是重复同一信息）？
- [ ] 如果本碎片是某暗线的"引爆点"（如0004是A线引爆点），前序碎片是否积累了足够的疑问/伏笔使引爆有冲击力？

**2.3 跨碎片物品/人物关联**

- [ ] 本碎片中出现的物品/符号/术语，是否在前置碎片中已有伏笔？
- [ ] 本碎片中出现的NPC（如果有跨碎片NPC），其行为/状态/记忆是否与前序出场一致？
- [ ] 本碎片是否为后续碎片播种了可追踪的线索？
- [ ] 检查"笔迹"（手写便笺、光墙刻字等）的连续性——同一人在不同碎片中的文字是否风格一致？

**2.4 氛围与情感递进**

- [ ] 本碎片的情感调性（困惑/悲伤/温暖/紧张/…）与前一个碎片相比，是否存在合理的递进或对比？
- [ ] 连续两个碎片是否情感调性过于相似（导致疲劳）或反差过大（导致断裂）？

**2.5 如本碎片为第一个碎片（无前置）**

- [ ] 本碎片作为"玩家第一印象"，教学引导是否充分？
- [ ] 核心概念（碎片/源印/溯光者/…）是否在碎片中以不突兀的方式被引入？
- [ ] 信息密度是否可控（不会在第一个碎片就灌输过多概念）？

**输出格式**：
```
| 衔接点 | 前置碎片 | 本碎片 | 状态 | 详情 |
|--------|---------|--------|------|------|
| 暗线A递进 | 0001 | 0002 | ✅ | … |
| 跨碎片物品 | 0001 | 0002 | ⚠️ | … |
| 情感递进 | … | … | … | … |

综合评级：🟢/🟡/🔴
```

### 维度 3：暗线植入审查

> 目标：检查三条暗线在当前碎片中的植入是否充分、准确、不突兀。

**3.1 植入点清单**

逐条列出碎片中所有暗线植入点：

| 暗线 | 植入位置（场景/对话节点） | 承载方式 | 信息内容 | 玩家感知度 |
|------|------------------------|---------|---------|-----------|
| A | … | NPC台词 | "…" | 明显/可忽略/仅二周目可见 |
| B | … | 环境道具 | "…" | … |
| C | … | 文本 | "…" | … |

**3.2 植入质量**

- [ ] 每条暗线的植入是否**不突兀**——即不打断当前场景的叙事节奏？
- [ ] 植入信息是否与《12碎片叙事规划》中为该碎片分配的暗线角色一致？（例：0004是A线引爆点，则必须在此处有足够强度的A线信息）
- [ ] 非主导暗线的"种子"植入是否足够隐晦，不至于喧宾夺主？
- [ ] 是否有任何暗线在本碎片中**完全缺席**？（检查三条暗线是否至少有一条在本碎片中有植入）

**3.3 暗线信息一致性**

- [ ] 本碎片暗线植入的信息是否与前置碎片中的同类信息一致？（例：B线中"冥府协议"的名称在不同碎片中是否一致）
- [ ] 本碎片的暗线信息是否与12碎片规划中的"暗线全景图"匹配？

**输出格式**：
```
| 暗线 | 植入点数 | 角色(种子/推进/引爆) | 质量评估 | 问题 |
|------|---------|---------------------|---------|------|
| A    | 2       | 推进                | 🟢      | —    |
| B    | 1       | 种子                | 🟡      | 仅在环境道具中出现，缺少NPC对话中的暗示 |
| C    | 0       | —                   | 🔴      | 完全缺席 |

综合评级：🟢/🟡/🔴
```

### 维度 4：剧情元素可实现性

> 目标：评估碎片中的场景、NPC行为、特殊效果在当前/计划技术栈下是否可实现。

**4.1 逐元素可实现性评估**

将碎片中的**每个关键视觉/交互/系统元素**列出，评估技术实现难度：

| 元素 | 类型 | 依赖系统 | 难度 | 降级方案 |
|------|------|---------|------|---------|
| … | 场景Shader | Shader | 🟢/🟡/🔴 | 用ColorRect替代 |
| … | NPC纸偶动画 | 美术资产 | 🟡 | 暂时使用静态占位 |
| … | 特殊UI效果 | UI系统 | … | … |

**判定标准**：
- 🟢 低风险：当前技术栈直接支持，无需新系统
- 🟡 中风险：需要新开发辅助系统，或需要尚未制作的美术资产
- 🔴 高风险：需要突破当前技术栈能力，或美术资产量远超当前资源

**4.2 NPC复杂度评估**

对每个NPC评估：
- [ ] 纸偶姿态数量：idle + 特殊动作 + 觉醒/转折姿态 = 总计？是否在美术产能范围内？
- [ ] 行为逻辑复杂度：简单（固定位置+单循环动画）/ 中等（条件触发+路径巡逻）/ 复杂（多阶段状态机+独立AI）
- [ ] 对话复杂度：对话节点数量 + 分支数量。LLM RAG 系统是否能承载？

**4.3 交互系统需求**

- [ ] 本碎片是否需要新的交互系统？（如：镜面反射、重力翻转、时间回溯…）
- [ ] 如果需要新系统，是否与现有代码架构兼容？
- [ ] 新系统的开发成本预估（小时计）

**输出格式**：
```
| 元素 | 可实现性 | 风险 | 降级方案 |
|------|---------|------|---------|
| … | 🟢/🟡/🔴 | … | … |

高风险元素总数：N
综合评级：🟢/🟡/🔴
```

### 维度 5：全局一致性审查

> 目标：检查本碎片的设计是否与其它文档的设定、数值、概念一致。

**5.1 跨文档数值一致性**

- [ ] 碎片ID、名称、难度标注：在所有文档中是否一致？
- [ ] NPC数量、名称：设计文档 vs NPC文档 vs 代码是否一致？
- [ ] 源印名称、形态、位置：设计文档 vs 12碎片规划是否一致？
- [ ] 修复进度百分比：是否与碎片总数对应的计算一致？
- [ ] 颜色-情感映射（如涉及0762的六色系统）：是否与全局定义一致？

**5.2 概念一致性**

- [ ] "溯光者"的身份定义在不同文档中的描述是否一致？
- [ ] "源印"的获取方式（手动推理发现 vs 自动触发）在不同碎片中是否统一？
- [ ] "碎片"的本质（AI意识投影 vs 公司培训沙盒 vs …）在不同文档中是否一致？
- [ ] NPC跨碎片出现时，其角色身份是否自洽？

**5.3 时间线一致性**

- [ ] 本碎片在12碎片时间线上处于哪个位置？与前后碎片的时间距离是否合理？
- [ ] 如果碎片涉及"零时之前/之后"的时间概念，其描述是否与世界观文档一致？

**5.4 与12碎片叙事规划比对**

逐项检查本碎片的设计是否与《12碎片叙事规划》中为该碎片定义的内容一致：

| 规划中的定义 | 本碎片实际设计 | 一致？ |
|-------------|--------------|--------|
| 世界名称 | … | ✅/❌ |
| 解密类型 | … | ✅/❌ |
| 暗线角色 | … | ✅/❌ |
| 警觉模式 | … | ✅/❌ |
| NPC数量 | … | ✅/❌ |
| … | … | … |

**输出格式**：
```
发现不一致项：
| # | 类型 | 文档A | 文档B | 不一致内容 | 建议 |
|---|------|-------|-------|-----------|------|
| 1 | 数值 | … | … | … | … |

不一致项总数：N
综合评级：🟢/🟡/🔴
```

### 维度 6：设计-代码对齐

> 目标：（仅在提供了代码时进行）检查代码实现是否与设计文档一致。

**6.1 核心逻辑对齐**

- [ ] 解密判定逻辑：代码中的条件判断是否与设计文档描述一致？
- [ ] 状态变量：代码中维护的状态变量是否覆盖设计文档中的所有状态？
- [ ] 触发条件：代码中的颜色/事件/阶段触发条件是否与设计文档一致？
- [ ] 依赖链：代码中的依赖关系（如红→黄）是否与设计文档一致？

**6.2 流程对齐**

- [ ] 碎片的开始条件、通关条件、失败条件在代码中是否正确实现？
- [ ] 源印的获取流程（位置+交互方式+显现方式+解码方式）在代码中是否正确？
- [ ] NPC行为阶段机（如老画家的三阶段对话）在代码中是否完整覆盖？

**6.3 未实现项**

- [ ] 列出设计文档中有但代码中未实现的功能
- [ ] 评估每项未实现功能的优先级（P0/P1/P2）

**6.4 代码健壮性**

- [ ] 是否有空指针/节点未找到的防护？
- [ ] 存档/读档后状态是否正确恢复？
- [ ] 是否存在"状态残留在跨场景后导致bug"的风险？

**输出格式**：
```
对齐情况：
| 功能 | 设计文档 | 代码 | 对齐？ |
|------|---------|------|--------|
| … | … | … | ✅/❌ |

未实现项：
| # | 功能 | 优先级 |
|---|------|--------|
| 1 | … | P1 |

综合评级：🟢/🟡/🔴
```

### 维度 7：玩家体验安全审查

> 目标：识别可能导致玩家无法继续游戏（软锁）、体验崩溃的认知过载、或情感疲劳的风险。

**7.1 软锁/死档风险**

- [ ] 是否存在"某种操作序列导致碎片永久无法完成"的路径？
- [ ] 如果玩家的警觉度达到最高等级（SHUTDOWN/LOCKDOWN），是否有恢复机制？
- [ ] 如果关键道具被使用后无法再次获取，是否有备份获取途径？
- [ ] 如果谜题允许无限次尝试但每次尝试消耗资源（警觉度/时间），是否有净消耗→软锁的风险？
- [ ] 存档/读档是否能从任何软锁状态中恢复？

**7.2 认知负荷**

- [ ] 本碎片新引入的概念/机制数量？（建议 ≤5 个全新概念）
- [ ] 本碎片同时需要玩家关注的信息维度？（解密+暗线+警觉+…，建议 ≤4 个并行关注点）
- [ ] 教学关卡（0001）的信息密度是否过高？
- [ ] 复杂碎片（如3015）之前是否有"准备提示"让玩家调整心理预期？

**7.3 情感节奏**

- [ ] 本碎片的情感调性是否与前后碎片形成合理弧线？
- [ ] 是否存在连续 3+ 个碎片都是"沉重/悲伤"主题的风险？（情感过载）
- [ ] 是否存在"关键情感时刻被过场动画/Tutorial打断"的风险？

**7.4 失败友好性**

- [ ] 谜题是否提供渐进式提示（而非"卡关5分钟后突然给答案"）？
- [ ] 如果玩家做出了"错误"的情感选择（如0002留下了不同的人），是否存在"正确"答案？
- [ ] 如果不希望有"正确"答案，不同分支的情感收束是否都具有叙事的完整性？

**输出格式**：
```
| 风险类型 | 风险点 | 严重度 | 建议 |
|---------|--------|--------|------|
| 软锁 | … | 🔴 | … |
| 认知负荷 | … | 🟡 | … |

综合评级：🟢/🟡/🔴
```

## 五、审查判定标准参考

### 评级含义

| 评级 | 含义 | 后续行动 |
|------|------|---------|
| 🟢 通过 | 该维度无问题或仅有可忽略的微小问题 | 无需修改 |
| 🟡 基本通过 | 存在 1-3 个需澄清或轻度调整的问题 | 在下一轮迭代中修复 |
| 🔴 存在关键问题 | 存在 ≥1 个会导致逻辑矛盾、玩家卡死、或系统性不一致的问题 | 当前版本不可发布，必须先修复 |

### 问题优先级

| 优先级 | 定义 | 例子 |
|--------|------|------|
| **P0** | 阻塞性问题：不修复则碎片不可玩或严重破坏体验 | 解密逻辑矛盾导致谜题不可解；软锁风险导致玩家无法通关 |
| **P1** | 重要问题：影响叙事质量或开发效率，应在当前碎片开发周期内修复 | 暗线缺失；文档间数值不一致；关键功能未实现 |
| **P2** | 优化建议：可延后但不修复会降低品质 | 美术降级方案确定；额外提示优化；文档局部澄清 |

## 六、Agent 审查工作流

当 Agent 收到本审查需求和剧情材料后，应按以下流程执行：

```
第1步：读取全部材料
  ├─ 必选材料（4份）全部读取
  └─ 可选材料按可用性读取

第2步：建立上下文
  ├─ 确认当前碎片在12碎片中的位置
  ├─ 确认该碎片的三条暗线角色
  ├─ 确认技术栈现状（从开发进度文档）
  └─ 列出前置碎片的关键信息（暗线状态/跨碎片物品/未解悬念）

第3步：逐维度审查（7个维度）
  ├─ 每个维度先列出所有检查项
  ├─ 逐项检查并记录结论
  └─ 给出该维度的综合评级

第4步：汇总与优先级
  ├─ 汇总所有维度的问题
  ├─ 按 P0/P1/P2 排序
  └─ 生成修复路线图

第5步：输出审查报告
  ├─ 按 §四 的格式生成报告文件
  └─ 保存至 design/review/ 目录
```

## 七、常见问题与特殊场景

### 7.1 如果本碎片是第一个碎片（无前置）
- 维度2"前置剧情衔接"改为审查"作为开篇碎片的教学引导质量"
- 重点关注：核心概念引入是否自然、信息密度是否可控、玩家第一印象设计

### 7.2 如果本碎片尚未有任何代码
- 跳过维度6"设计-代码对齐"
- 维度4"可实现性"改为纯技术可行性评估（基于当前技术栈）

### 7.3 如果本碎片是纯叙事碎片（无解密）
- 维度1"解密逻辑合理性"改为"叙事逻辑合理性"
- 审查焦点：叙事是否有内在矛盾、角色行为是否自洽、因果关系是否成立

### 7.4 如果审查一组连续碎片
- 维度2扩展到检查整组的内部衔接
- 额外审查整组的暗线递进节奏
